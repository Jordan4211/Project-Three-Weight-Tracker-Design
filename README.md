## Mobile App Portfolio Artifact: Weight Tracker

This artifact is a fully functional Android weight-tracking app built in Java in Android Studio. The app lets users create an account, log in, save daily weight entries, update and delete entries, set a goal weight, and optionally enable SMS alerts when they reach that goal. The main goal of the app was to give users a simple way to track weight over time. It was designed for users who want a lightweight tool for daily logging, goal tracking, and basic progress monitoring.

The app needed a few core screens to support those user needs. The login screen lets new users create an account and returning users sign in. The weight history screen shows saved entries in a grid and gives users a way to add, edit, and delete data. The add weight screen handles new entries, the goal screen lets users save a target weight, and the SMS screen handles permission requests and alert setup. The UI was built to keep the main tasks obvious. Each screen had one main job, the buttons were placed where users would expect them, and the flow between screens matched how someone would actually use the app.

I coded the app in parts. I built the UI first so I knew exactly what I had to connect. After that I added the SQLite database, then the login logic, then CRUD for weight entries, then the goal check, and then SMS permission and SMS sending. I kept the work split into small pieces and tested after each change. That made debugging easier and kept problems from stacking up. This same method would work well in future projects because it gives structure to the build process and makes it easier to isolate bugs.

Testing was a big part of the project. I used the Android Emulator to test each screen after changes. I checked account creation, login, database persistence, adding entries, editing entries, deleting entries, saving a goal, and both SMS permission paths. This mattered because a feature can look finished but still break once it interacts with the database or another screen. Testing revealed issues with navigation, missing IDs, manifest setup, Gradle configuration, and how permissions behaved when denied. Catching those early saved time later.

The biggest challenge was not the UI. It was wiring everything together without breaking something else. Login, SQLite, CRUD, and SMS alerts all depend on each other. I had to think through what happens when the user is new, when data is empty, when permission is denied, and when the goal is reached more than once. The main workaround was to keep the code split by responsibility and build the app one layer at a time instead of trying to finish everything at once.

The part of the project where I showed the most growth in my opinion was the full app flow from login to stored data to goal checking. That part pulled together UI design, persistence, validation, CRUD, and permissions into one working app. It was the strongest proof that I can take an app from planning to a functional final version instead of just designing screens.
